# Bug Fixes - December 8, 2025

## Overview

This document covers three bug fixes implemented to improve admin functionality and data consistency.

## 1. Timeline Rebuild Script

### Problem
The homepage "Latest Activity" section displayed mock/hardcoded data that didn't exist in the database, showing incorrect projects and publications.

### Root Cause
The `ContentTimeline` table contained stale entries that were not synchronized with actual content in the database.

### Solution
Created a utility script `/Users/aravind/Projects/portfolio/backend/scripts/rebuild-timeline.ts` that:
1. Clears all existing timeline entries
2. Rebuilds from actual database content:
   - Published projects
   - All publications
   - Published guidebooks
3. Properly maps tags, dates, and metadata

### Implementation Details

**Script Location**: `backend/scripts/rebuild-timeline.ts`

**Execution**:
```bash
cd backend
npx ts-node scripts/rebuild-timeline.ts
```

**Key Functions**:
- Uses Prisma transaction to ensure atomic operation
- Rebuilds timeline in order: projects → publications → guidebooks
- Maintains proper relationships with tags
- Reports count of items added

**Output Example**:

## 2. Guidebook Draft Visibility Fix

### Problem
Draft guidebooks created in the admin panel were not visible in the admin guidebooks list, making them impossible to edit or publish.

### Root Cause
The `getGuidebooks()` service method was hardcoded to filter by `published: true`, preventing draft guidebooks from being returned even when requested by the admin panel.

### Solution
Made the `published` filter optional and conditional:
1. Added `published` as optional parameter to query schema
2. Modified service to only apply published filter when explicitly provided
3. Admin panel passes empty object `{}`, retrieving all guidebooks
4. Public pages can still filter by passing `{ published: true }`

### Implementation Details

**Files Modified**:
1. `backend/src/schemas/guidebook.schema.ts:27`
   - Added `published: z.coerce.boolean().optional()` to `guidebookQuerySchema`

2. `backend/src/services/guidebook.service.ts:17-31`
   - **Before**:
     ```typescript
     const where: Prisma.GuidebookWhereInput = {
       published: true,
       ...(featured !== undefined && { featured }),
     };
     ```
   - **After**:
     ```typescript
     const where: Prisma.GuidebookWhereInput = {
       ...(published !== undefined && { published }),
       ...(featured !== undefined && { featured }),
     };
     ```

**Behavior**:
- Admin panel: `useGuidebooks({})` → Returns ALL guidebooks (drafts + published)
- Public pages: `useGuidebooks({ published: true })` → Returns only published
- Filtering: `useGuidebooks({ published: false })` → Returns only drafts

### Testing
- Create draft guidebook in admin panel
- Verify it appears in admin guidebooks list
- Verify it shows "Draft" badge
- Verify it doesn't appear on public guidebooks page

## 3. Publication Date/Time Picker Fix

### Problem
The date and time picker in the publication form displayed random numbers when clicking on the calendar or time controls.

### Root Cause
Format mismatch between form state and HTML5 `datetime-local` input:
- Form stored ISO 8601 format: `2025-12-08T10:30:00.000Z`
- Input expects: `2025-12-08T10:30`
- Code was slicing ISO string: `.slice(0, 16)`, which created invalid format

### Solution
Properly convert between formats using a helper function:
1. Store datetime in `datetime-local` format in form state
2. Convert to ISO only when submitting to API
3. Convert from ISO when loading existing publication

### Implementation Details

**File Modified**: `frontend/src/components/admin/PublicationForm.tsx`

**Changes**:

1. **Added Helper Function** (line 21-30):
   ```typescript
   const toDateTimeLocal = (isoString: string) => {
     const date = new Date(isoString);
     const year = date.getFullYear();
     const month = String(date.getMonth() + 1).padStart(2, '0');
     const day = String(date.getDate()).padStart(2, '0');
     const hours = String(date.getHours()).padStart(2, '0');
     const minutes = String(date.getMinutes()).padStart(2, '0');
     return `${year}-${month}-${day}T${hours}:${minutes}`;
   };
   ```

2. **Updated Form Initialization** (line 38):
   ```typescript
   publishedAt: toDateTimeLocal(new Date().toISOString()),
   ```

3. **Updated useEffect** (line 57):
   ```typescript
   publishedAt: toDateTimeLocal(publication.publishedAt || new Date().toISOString()),
   ```

4. **Simplified Input** (line 199-207):
   ```typescript
   <Input
     id="publishedAt"
     type="datetime-local"
     value={formData.publishedAt}
     onChange={(e) => {
       setFormData({ ...formData, publishedAt: e.target.value });
     }}
     required
   />
   ```

5. **Convert on Submit** (line 82):
   ```typescript
   const submitData = {
     ...formData,
     publishedAt: new Date(formData.publishedAt).toISOString(),
     // ... rest of data
   };
   ```

### Data Flow

**Before Fix**:
```
Form State: ISO string
         ↓
Input: .slice(0, 16) → Invalid format → Random numbers
```

**After Fix**:
```
Load:    ISO → toDateTimeLocal() → Form State
Display: Form State → Input (valid format)
Change:  Input → Form State (datetime-local format)
Submit:  Form State → toISOString() → API (ISO format)
```
