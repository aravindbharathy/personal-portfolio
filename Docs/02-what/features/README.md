# Features Catalog

This directory contains detailed descriptions of major features in the portfolio website.

## Purpose

Feature documents provide high-level descriptions of what each feature does and why it exists. They answer:
- What does this feature do?
- Who uses it?
- Why is it valuable?
- How does it fit into the product?

## Features in This Catalog

1. **[Projects](./projects.md)** - Research case study portfolio with detailed documentation
2. **[Publications](./publications.md)** - Multi-platform article aggregation and display
3. **[Guidebooks](./guidebooks.md)** - Curated thematic collections of articles
4. **[Admin Panel](./admin-panel.md)** - Content management and publishing controls
5. **[Authentication](./authentication.md)** - Secure admin access system

## Feature vs. Spec Distinction

### These Are Features (High-Level)
- Describe user-facing capabilities
- Explain business value
- No technical implementation details
- No acceptance criteria

### Specs Are Different (see `03-how/specs/`)
- Detailed requirements with acceptance criteria
- Testable success conditions
- Out-of-scope clarifications
- Written before implementation

## When to Create a Feature Document

✅ **Create when:**
- Feature is a core differentiator
- Feature is complex with multiple sub-capabilities
- New team members need product context
- Feature has significant user value

❌ **Don't create when:**
- Feature is simple/obvious (basic UI elements)
- Feature is purely infrastructure (database indexes)
- Feature is adequately covered in core-product.md
- Feature is a minor utility
